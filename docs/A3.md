### **Objetivos y Requerimientos.**

### 

### **Objetivo General**

Desarrollar una plataforma centralizada de gestión y visualización de datos de emergencias para optimizar la toma de decisiones, reducir los tiempos de respuesta y mejorar la coordinación de los cuerpos de bomberos urbanos.

### **Objetivos Específicos**

* **OE1:** Implementar un sistema de información geoespacial que permita visualizar en tiempo real la ubicación de incidentes, unidades de emergencia y recursos clave sobre un mapa interactivo.  
* **OE2:** Crear una base de datos robusta y escalable, poblada con datos sintéticos realistas, para almacenar y consultar el historial de emergencias, facilitando el análisis post-incidente.  
* **OE3:** Desarrollar un panel de control (dashboard) que presente métricas y estadísticas relevantes a partir de los datos históricos, permitiendo identificar patrones, tendencias y áreas de alta incidencia.  
* **OE4:** Empaquetar la solución web en una aplicación móvil para Android, garantizando el acceso a la información crítica desde dispositivos móviles en terreno.

**Requerimientos.**

### Nivel Operacional

| Código | Requerimiento |
|:------:|:-------------|
| **RF01** | El sistema deberá cargar una **vista de mapa predeterminada** al iniciar, centrada en un área geográfica de interés definida. |
| **RF02** | El sistema deberá presentar un **mapa geoespacial interactivo** como componente principal de la interfaz de usuario. |
| **RF03** | El usuario deberá poder realizar operaciones de **zoom y desplazamiento (pan)** sobre el mapa para explorar diferentes áreas geográficas. |
| **RF04** | El sistema deberá renderizar los incidentes como **marcadores geo-referenciados** en la capa principal del mapa. |
| **RF05** | Al interactuar con un marcador de incidente, el sistema deberá desplegar una **ventana emergente (popup)** con información sumaria: ID, tipo y fecha del evento. |
| **RF06** | El sistema deberá proveer acceso a una **vista de detalle completa** del incidente, que contenga todos los atributos asociados al registro. |
| **RF07** | El sistema deberá permitir el **filtrado de incidentes** visibles en el mapa mediante un rango de fechas (inicio y fin) especificado por el usuario. |
| **RF08** | El sistema deberá permitir el **filtrado de incidentes** por su tipología (ej. incendio estructural, accidente vehicular). |
| **RF09** | El sistema deberá permitir el **filtrado de incidentes** según su nivel de severidad (ej. bajo, medio, alto). |
| **RF10** | El sistema deberá soportar la **combinación de múltiples criterios de filtrado** de forma simultánea para refinar las consultas de datos. |
| **RF11** | El sistema deberá incluir una funcionalidad de **búsqueda por identificador único (ID)** para la localización rápida de un incidente específico. |
| **RF12** | El sistema deberá presentar la información de los incidentes en una **vista de tabla estructurada**. |
| **RF13** | La tabla de incidentes deberá implementar un **mecanismo de paginación** para gestionar eficientemente grandes volúmenes de datos. |
| **RF14** | El usuario podrá **ordenar los registros** de la tabla de incidentes de forma ascendente o descendente basándose en sus columnas (ej. fecha, tipo). |
| **RF15** | El sistema deberá disponer de un **formulario para la creación** de nuevos registros de incidentes, accesible a usuarios con permisos. |
| **RF16** | El formulario de creación deberá incluir campos específicos para **tipo, fecha, hora, ubicación geoespacial (latitud/longitud) y descripción**. |
| **RF17** | El sistema deberá aplicar **reglas de validación de datos** en el formulario de creación para asegurar la integridad de la información. |
| **RF18** | El sistema deberá mostrar una **leyenda cartográfica** que describa la simbología utilizada para representar los diferentes elementos en el mapa. |
| **RF19** | El sistema deberá permitir al usuario cambiar la capa base del mapa, ofreciendo **como mínimo dos alternativas de visualización** (ej. mapa de calles y vista satelital). |
| **RF20** | El sistema deberá visualizar la ubicación de las **estaciones de bomberos** como una capa de íconos fijos en el mapa. |

| **RF38** | El sistema deberá presentar un **indicador (KPI)** del porcentaje de incidentes atendidos dentro de un umbral de tiempo de respuesta objetivo. |
| **RF39** | El sistema deberá permitir la **comparación del volumen de incidentes** entre el trimestre actual y el mismo trimestre del año anterior. |

| **RF41** | El sistema deberá generar un reporte que identifique el **tipo de incidente más frecuente por cada distrito** o comuna. |
| **RF42** | El sistema deberá mostrar una **proyección lineal simple** del número de incidentes esperados para el próximo período, basada en datos históricos. |
| **RF43** | El sistema deberá permitir la **comparación de rendimiento** (ej. volumen de incidentes atendidos) entre diferentes estaciones de bomberos. |
| **RF44** | El usuario podrá visualizar la **tendencia de un tipo de incidente específico** a lo largo del tiempo para identificar patrones de crecimiento o decrecimiento. |

### Requisitos No Funcionales (RNF)

| Código | Requerimiento |
|:------:|:-------------|
| **RNF01 (Rendimiento)** | La aplicación debe ser capaz de renderizar en el mapa hasta 5,000 puntos de datos simultáneamente en menos de 5 segundos. Las consultas y filtros sobre el conjunto de datos completo deben resolverse en un promedio de menos de 2 segundos. |
| **RNF02 (Usabilidad)** | La interfaz de usuario debe ser intuitiva, permitiendo que un usuario sin conocimientos técnicos avanzados pueda operar todas las funciones operacionales y tácticas con menos de 30 minutos de capacitación. El diseño se adherirá a principios de usabilidad y claridad visual. |
| **RNF03 (Compatibilidad)** | La aplicación web deberá ser compatible con las últimas dos versiones estables de los navegadores Google Chrome, Mozilla Firefox y Microsoft Edge, asegurando una experiencia de usuario consistente entre plataformas. |
